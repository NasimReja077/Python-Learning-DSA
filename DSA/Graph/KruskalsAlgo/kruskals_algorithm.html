What is Kruskal's — and How is it Different from Prim's?

Real World Analogy

Imagine you have a list of all possible roads between cities, each with a construction cost. You sort the list from cheapest to most expensive. Then you go down the list — build each road if it connects two previously unconnected regions. Skip it if both cities are already connected (building it would make a loop). Stop when all cities are in one connected region.

KRUSKAL'S (this page)

Edge-centric: thinks globally about all edges at once
Step 1: sort ALL edges by weight
Step 2: greedily add edges, skip if they form a cycle
Cycle detection: Union-Find in O(α(V)) ≈ O(1)
Best for: sparse graphs (E << V²)
Complexity: O(E log E)

PRIM'S (for comparison)

Vertex-centric: grows one connected tree from a start node
Step 1: pick a start vertex
Step 2: always expand to the cheapest adjacent vertex
Min tracking: min-heap or linear scan
Best for: dense graphs (E ≈ V²)
Complexity: O(E log V) with heap
📋

Sort First

All edges sorted by weight ascending. This is the O(E log E) step. After sorting, we process them in order — cheapest first.

🌲

Forest → Tree

Starts with V separate trees (one per vertex). Each union operation merges two trees. Finishes when there's exactly one tree.

🔗

Cycle = Same Component

An edge creates a cycle if and only if both its endpoints are already in the same component. Union-Find checks this instantly.

🛑

Stop at V−1 Edges

A spanning tree for V vertices needs exactly V−1 = 4 edges. Once we've added that many, the MST is complete.

02

Union-Find (DSU) — The Secret Weapon

What Problem Does Union-Find Solve?

We need to answer one question very fast: "Are vertices u and v in the same connected component?"

Without DSU: scan the whole graph — O(V+E). Too slow if asked millions of times.
With DSU: O(α(V)) ≈ almost O(1), where α is the inverse Ackermann function (effectively ≤ 5 for any realistic input).

find(x)
returns the ROOT of x's component
Follows the parent chain from x upward until it reaches a node that is its own parent (the root). Includes path compression: after finding the root, it makes every visited node point directly to the root, flattening the tree for future calls.
# Before find(4):
# 4 → 3 → 1 → 1 (root)
# After path compression:
# 4 → 1, 3 → 1 (flattened!)
union(x, y)
merges two components into one
First calls find(x) and find(y). If same root → they're already connected → skip (would create cycle). Otherwise merges using union by rank: attach the shorter tree under the taller one to keep trees flat and efficient.
# rank[root_x] > rank[root_y]?
# → parent[root_y] = root_x
# Equal rank?
# → parent[root_y] = root_x, rank[root_x]++
DSU Arrays — Initial State (each vertex is its own component)
parent[i] = i → vertex i is its own root (5 separate components)
rank[i] = 0 → all trees start at height 0
03

Core Idea — 3 Steps, That's It

1
Sort all edges by weight ascending. For E=7 edges: (0-1,2), (1-2,3), (1-4,5), (0-3,6), (2-4,7), (1-3,8), (3-4,9). This single sort dominates the runtime at O(E log E).
2
Initialize DSU: parent[i]=i, rank[i]=0 for all i. Each vertex starts as its own isolated tree. We have V=5 components.
3
Process edges in sorted order. For each edge (u, v, w): call find(u) and find(v). If same root → SKIP (cycle). If different roots → ADD edge to MST, call union(u, v). Stop when MST has V−1=4 edges.

Why do we stop at edge #4?
A spanning tree of V vertices always has exactly V−1 = 4 edges. After adding edge #4 (0-3), all 5 vertices are connected in one component. Edges #5, #6, #7 are never even examined — we can exit early.

kruskal.py — Union-Find (DSU) class 
 1class UnionFind:
 2  def __init__(self, n):
 3    self.parent = list(range(n))   # parent[i]=i → each is its own root★ Init
 4    self.rank   = [0] * n           # rank[i]=0 → all trees have height 0
 5
 6  def find(self, x):               # returns root of x's component
 7    if self.parent[x] != x:        # x is not its own root★ Path Compression
 8      self.parent[x] = self.find(self.parent[x]) # recurse + compress
 9    return self.parent[x]          # return root
10
11  def union(self, x, y):           # merge components of x and y
12    rx, ry = self.find(x), self.find(y)
13    if rx == ry: return False       # same root = same component = CYCLE!★ Cycle check
14                                     # union by rank: attach smaller under larger
15    if   self.rank[rx] <  self.rank[ry]: self.parent[rx] = ry
16    elif self.rank[rx] >  self.rank[ry]: self.parent[ry] = rx
17    else:  self.parent[ry] = rx; self.rank[rx] += 1★ Union by Rank
18    return True                      # successfully merged — edge is safe

kruskal.py — Path Compression visualized (why it matters) 
# WITHOUT path compression — tree can become a chain:
#
#   parent = [0, 0, 1, 2, 3]
#   find(4): 4→3→2→1→0  (4 hops)
#   find(4): 4→3→2→1→0  (4 hops again — nothing saved)
#
# WITH path compression — tree flattens to a star:
#
#   find(4): 4→3→2→1→0  (first call: 4 hops)
#   parent becomes: [0, 0, 0, 0, 0]  (all point to root!)
#   find(4): 4→0  (next call: 1 hop!)
#
# Combined with union-by-rank, this gives amortized O(α(n))
# α = inverse Ackermann. For n ≤ 10^80 (more atoms than universe): α ≤ 5
# Effectively: O(1) per operation

O(E log E) vs O(E log V) — Are They the Same?
Yes, essentially. Since E ≤ V², we have log(E) ≤ log(V²) = 2·log(V).
So O(E log E) = O(E log V) — they're equivalent within a constant factor.

For this graph: E=7, log(7) ≈ 2.8 → Kruskal's does about 20 significant operations.
Compare to matrix-based Prim's: V² = 25 operations — Kruskal wins on sparse graphs.

Same MST as Prim's — Is That Always the Case?
Yes, when edge weights are unique. If all edge weights are distinct, there is exactly one MST, so any correct algorithm must find it.

When weights tie: multiple valid MSTs may exist. Prim's and Kruskal's may pick different ones — both are correct (same total weight, different edges). The exact MST found depends on tie-breaking order.
